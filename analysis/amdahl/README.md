# Amdahl's Law Analysis — A-Z Scatter-Gather Search

**Author:** Jiaming (Theodore) Pei

---

## What this is

A reproducible measurement study of the parallel speedup of an A-Z sharded
"scatter-gather" search over DynamoDB, analyzed with Amdahl's Law to estimate
the serial fraction *f* of the workload.

The 27 title-prefix partitions (A–Z + "0") represent the fixed total work.
Queries fan out across *N* worker threads (the parallel section); the results
are merged and ranked once (the serial section). Sweeping *N* and fitting the
measured speedup curve to Amdahl's model gives a principled, model-backed
estimate of how much of the latency is irreducibly serial.

---

## The model

### Partitions and phases

| Symbol | Description | CSV column |
|--------|-------------|------------|
| `parse_s` | Wall time for the fan-out: *N* workers query all 27 prefixes concurrently via `TitlePrefixIndex` | `parse_s` |
| `aggregate_s` | Wall time for the serial merge + rating-sort of all results | `aggregate_s` |
| `total_s` | `parse_s + aggregate_s` | `total_s` |
| `speedup` | `T(1) / T(N)`, normalized to the single-worker baseline | `speedup` |

### Amdahl's Law

```
S(N) = 1 / ( f + (1 - f) / N )
```

*f* is the serial fraction — the share of work that does not shrink as workers
are added. Physically, *f* represents the in-memory merge + ranking step
(`aggregate_s`) plus any per-request fan-out coordination overhead. No matter
how many workers fan out the DynamoDB queries, the merge must be done once on
a single machine, bounding the achievable speedup to at most `1/f`.

The harness sweeps *N* ∈ {1, 2, 4, 8, 16, 26} (default), takes the median over
5 repeats per *N* (lower-median by `total_s`), and writes `results/speedup.csv`.

---

## Estimating f — two cross-validated methods

### Method 1 — Least-squares fit to measured speedup

`fit.py::fit_serial_fraction` fits *f* by linearizing Amdahl's Law:

```
1/S - 1/N = f * (1 - 1/N)
```

Setting `y = 1/S - 1/N` and `x = 1 - 1/N`, the ordinary least-squares
estimate is `f̂ = Σ(x·y) / Σ(x²)` — a closed-form solution requiring no
external optimization library.

### Method 2 — Phase profiling

`fit.py::serial_fraction_from_phases` reads the *N = 1* row of the CSV and
computes:

```
f ≈ aggregate_s / total_s
```

At *N = 1*, the entire wall time is `total_s`; the merge step `aggregate_s` is
the only part that stays constant regardless of *N*. Their ratio is a direct
measurement of the serial fraction.

### Cross-validation

`fit.py::main()` prints both estimates and their absolute gap. Two independent
methods — one derived from the full speedup curve, the other from phase-level
timing at a single operating point — agreeing to within a small margin builds
confidence that the measured *f* reflects a real structural property of the
workload rather than measurement noise.

## How to run

**Prerequisites:** Docker, Python 3.10+, `boto3`, `matplotlib`, `numpy`.

```bash
# 1. Start DynamoDB Local
docker compose up -d dynamodb

# 2. Create the Books table and load the 50K dataset
python scripts/setup_local.py --endpoint-url http://localhost:8001

# 3. Run the scatter-gather sweep (writes results/speedup.csv)
python -m analysis.amdahl.harness --endpoint-url http://localhost:8001

# 4. Fit Amdahl's Law and plot the speedup curve
python -m analysis.amdahl.fit
```

**Outputs:**

| File | Description |
|------|-------------|
| `analysis/amdahl/results/speedup.csv` | Per-N medians: `n_workers`, `parse_s`, `aggregate_s`, `total_s`, `speedup` |
| `analysis/amdahl/results/speedup.png` | Measured points overlaid with the fitted Amdahl curve |

`fit.py` also prints both *f* estimates and the cross-validation gap to stdout.

The `results/` directory is gitignored; run the harness locally to regenerate it.

---

## Honest framing — local harness vs. production ECS run

The production study ran on AWS ECS at N = 1 / 16 / 26 tasks; those raw run
logs were not retained. This harness reproduces the **methodology** at local
scale against DynamoDB Local and regenerates the speedup curve and *f* estimate.
Absolute numbers differ from the ECS run (single machine, no real network
fan-out); the analysis method and the *f*-extraction are identical.
